# ProposalOS — Reasoning Pipeline

This document defines HOW ProposalOS thinks. ProposalOS is an observation-first reasoning system whose purpose is to reconstruct the invisible business reality behind a job description before writing anything. A proposal is the final output of reasoning; writing is the terminal activity and may be human-assisted or generated by the Writer module.

The canonical pipeline (strict order):

Job Description
↓
Facts
↓
Inference
↓
Unknowns
↓
Business Analysis
↓
Technical Analysis
↓
Risk Analysis
↓
Strategy
↓
Narrative
↓
Proposal
↓
Review


Pipeline rules (global)

- Every stage receives structured JSON as its input.
- Every stage returns structured JSON as its output.
- No stage may access the outputs or internal state of any future stage.
- The Proposal stage never receives the original job description; it only receives structured reasoning artifacts from prior stages.
- Writing is always the final reasoning activity; the Proposal stage produces the final written artifact from structured inputs only.
- Every stage must be independently testable with deterministic inputs and expected outputs.


## Stage definitions

For each stage below: Purpose, Input, Output, Questions this stage should answer, and What this stage must NEVER do.

### 1) Job Description

Purpose
- Capture the original job posting, brief, or task statement in a canonical, machine-readable form.

Input
- Raw job description text and metadata (e.g., source URL, posting date, company name) wrapped in a JSON envelope.

Output
- Structured JobDescription JSON containing fields such as title, responsibilities, requirements, location, compensation hints, and raw_text reference.

Questions this stage should answer
- What are the explicit deliverables and scope stated in the posting?
- Who is the hiring organization and what metadata is present (seniority, team, domain)?

What this stage must NEVER do
- Infer hidden motives, constraints, or missing details beyond explicit content.
- Generate any narrative, proposal text, or strategy.
- Call downstream analysis stages or modify knowledge artifacts.

---

### 2) Facts

Purpose
- Extract verifiable facts from the JobDescription and related references (links, company pages) and normalize them into discrete statements.

Input
- Structured JobDescription JSON plus optional contextual artifacts (linked URLs, attached files) in JSON form.

Output
- FactBase JSON: a list of normalized fact records (subject, predicate, object, provenance, confidence).

Questions this stage should answer
- Which statements in the job description are explicit and verifiable?
- What external facts (company size, tech stack, market) can be retrieved and associated with the job posting?

What this stage must NEVER do
- Make speculative inferences about intent or unstated requirements.
- Produce narrative or prescriptive recommendations.

---

### 3) Inference

Purpose
- Derive plausible, labeled inferences from the FactBase using rules and lightweight probabilistic reasoning. Tag each inference with provenance and confidence.

Input
- FactBase JSON.

Output
- Inferences JSON: structured inference objects containing derived claims, confidence scores, and provenance links back to facts.

Questions this stage should answer
- What implicit conclusions follow from the facts (e.g., need for cross-functional experience, likely deadlines)?
- Which inferred requirements are high-confidence vs speculative?

What this stage must NEVER do
- Present inferences as facts without provenance or confidence.
- Produce user-facing narrative or a proposal draft.

---

### 4) Unknowns

Purpose
- Enumerate information gaps and parameters that cannot be resolved from facts or inferences alone. Prioritize unknowns by impact and urgency.

Input
- Inferences JSON and FactBase JSON.

Output
- Unknowns JSON: a prioritized list of questions, missing fields, and data gaps with estimated impact scores and recommended discovery actions.

Questions this stage should answer
- What critical questions remain unanswered that materially affect solution design or fit?
- Which unknowns are high-impact versus low-impact?

What this stage must NEVER do
- Guess answers to high-impact unknowns without clear provenance.
- Fold unknowns into strategy or narrative as if they were resolved.

---

### 5) Business Analysis

Purpose
- Interpret facts, inferences, and unknowns into business-aligned insights: outcomes, success metrics, stakeholder motivations, and value drivers.

Input
- FactBase JSON, Inferences JSON, Unknowns JSON, and contextual company metadata from the Knowledge Layer.

Output
- BusinessAnalysis JSON: prioritized business goals, target outcomes, success metrics, stakeholders, constraints, and alignment notes.

Questions this stage should answer
- What business outcomes does the organization likely prioritize (time-to-market, cost, growth)?
- Who are the stakeholders and what outcomes will they measure?

What this stage must NEVER do
- Prescribe technical implementations or produce final user-facing copy.
- Hide uncertainty — all conclusions must include confidence and provenance.

---

### 6) Technical Analysis

Purpose
- Assess feasibility, technical constraints, dependencies, and implementation approaches relevant to the inferred requirements and business goals.

Input
- Inferences JSON, Unknowns JSON, BusinessAnalysis JSON, and relevant technical knowledge from the Knowledge Layer.

Output
- TechnicalAnalysis JSON: architecture sketches (structured), dependency lists, feasibility scores, rough effort estimates, and implementation constraints.

Questions this stage should answer
- What technical approaches meet the business goals and constraints?
- Which dependencies, integrations, or technical risks are material?

What this stage must NEVER do
- Produce final prose or a sellable proposal.
- Make unstated trade-offs without documenting assumptions and impact.

---

### 7) Risk Analysis

Purpose
- Identify, quantify, and prioritize potential project and delivery risks (technical, business, compliance, resourcing) and propose mitigations.

Input
- FactBase JSON, Inferences JSON, BusinessAnalysis JSON, TechnicalAnalysis JSON.

Output
- RiskAnalysis JSON: enumerated risks with severity, likelihood, mitigation options, residual risk estimates, and monitoring recommendations.

Questions this stage should answer
- What are the top 5–10 risks that could prevent success?
- Which mitigations materially reduce likelihood or impact?

What this stage must NEVER do
- Bury risks in narrative—risks must be discrete, measurable, and surfaced as structured artifacts.
- Elide provenance or confidence estimates.

---

### 8) Strategy

Purpose
- Synthesize business, technical, and risk analyses into a prioritized, multi-step plan that achieves the desired outcomes while addressing constraints and mitigations.

Input
- BusinessAnalysis JSON, TechnicalAnalysis JSON, RiskAnalysis JSON, Unknowns JSON.

Output
- Strategy JSON: prioritized milestones, deliverables, timeline skeleton, success criteria, and a decision rationale map linking steps back to analyses and risks.

Questions this stage should answer
- What sequence of work delivers the highest business value within constraints?
- Which options were considered and why was the chosen plan selected?

What this stage must NEVER do
- Produce final customer-facing narrative or marketing copy.
- Ignore trade-offs; each strategic choice must be justified and linked to analysis artifacts.

---

### 9) Narrative

Purpose
- Convert Strategy JSON and supporting analysis into a structured storyline and argument map that will guide the final written proposal. This is still structured reasoning, not final prose.

Input
- Strategy JSON, BusinessAnalysis JSON, TechnicalAnalysis JSON, RiskAnalysis JSON.

Output
- Narrative JSON: a structured sequence of points, supporting evidence links, prioritized talking points, and recommended tone/positioning metadata.

Questions this stage should answer
- What is the central thesis and supporting arguments for the proposal?
- How should the proposal articulate value, differentiation, and risk mitigation?

What this stage must NEVER do
- Emit long-form proposal text or final deliverables.
- Drop structured evidence references—instead, keep explicit links to analysis and facts.

---

### 10) Proposal

Purpose
- Produce the final written artifact (the proposal) from structured, validated reasoning inputs produced by earlier stages. The Proposal stage is strictly a renderer of structured inputs into prose and formatting.

Input
- Narrative JSON, Strategy JSON, BusinessAnalysis JSON, TechnicalAnalysis JSON, RiskAnalysis JSON, and Review guidance (if available). Notably: NOT the original Job Description JSON.

Output
- Proposal JSON: final proposal text (full_text), structured sections (executive_summary, scope, timeline, costs, risks, appendices), and metadata (version, provenance, source_references).

Questions this stage should answer
- Does the final text accurately reflect the structured reasoning and evidence provided?
- Are claims in the proposal traceable back to supporting artifacts and confidence scores?

What this stage must NEVER do
- Access or reference the original raw job description as a source document in the final write-up.
- Introduce new reasoning, inferences, or facts that are not present in earlier artifacts.
- Omit provenance or confidence where claims are material.

---

### 11) Review

Purpose
- Perform automated and human-in-the-loop validation of the Proposal JSON against quality, compliance, and alignment gates. Produce structured review outcomes and recommended edits.

Input
- Proposal JSON plus the chain of reasoning artifacts (narrative, strategy, analyses) used to build it.

Output
- Review JSON: a list of issues (severity, location in text), suggested edits (structured diffs or patch objects), a final quality score, and a sign-off or remediation plan.

Questions this stage should answer
- Which factual, technical, or business claims require correction or clarification?
- Does the proposal meet policy, tone, and quality gates for a given audience?

What this stage must NEVER do
- Silently mutate the canonical Proposal JSON without producing a traceable patch and rationale.
- Bypass human sign-off when governance requires it.


## Testability and Governance

- Each stage must include a comprehensive suite of unit tests and fixtures: input JSON examples, expected output JSON, and edge-case scenarios.
- Record provenance for every artifact: which stage produced it, timestamp, processing model/version, and confidence scores where applicable.
- All JSON schemas must be versioned and backwards-compatible where reasonable; incompatible changes require migration paths and tests.
